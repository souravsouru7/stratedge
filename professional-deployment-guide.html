Every code change goes through this exact sequence before reaching real users. Nothing skips a step.

💻

Never write code on main or staging directly. Create a new branch for every change, no matter how small.

↓
push to GitHub
🐙

GitHub Actions automatically runs your tests the moment you open a PR. If any test fails, the PR is blocked — you cannot merge broken code.

↓
merged to staging branch

  • Same EC2, different folder
  • Separate MongoDB Atlas cluster

  • Only you access this

  • Real users, real data

  • Only gets tested code
  • .env.production file
  • PM2 manages 2 processes

Section 05

What Tests to Write

Don't write tests for everything — that's a waste of time. Write tests for the flows that break most often and hurt the most when they do.

Layer 1
Automated Tests
Jest + Supertest. Runs on every PR in GitHub Actions. Blocks bad code from merging. Catches broken APIs before deployment.
Layer 2
Manual Staging QA
5-minute checklist you run on staging before merging to main. Catches UI/UX regressions that automated tests can't see.
Layer 3
Sentry Monitoring
Already installed. Real-time error alerts for production. You get emailed instantly when something breaks with full stack trace.

Automated Tests — Start With These 5 Files

# File structure
backend/
  tests/
    auth.test.js          ← login, signup, JWT verify
    trade.test.js         ← upload endpoint, status check
    indianTrade.test.js   ← Indian market CRUD
    analytics.test.js     ← PnL calculations
    upload.test.js        ← file validation, Cloudinary mock
// Example: backend/tests/auth.test.js
const request = require('supertest');
const app = require('../server');

describe('POST /api/auth/login', () => {
  it('returns JWT with valid credentials', async () => {
    const res = await request(app)
      .post('/api/auth/login')
      .send({ email: 'test@test.com', password: 'password123' });

    expect(res.status).toBe(200);
    expect(res.body).toHaveProperty('token');
  });

  it('returns 401 with wrong password', async () => {
    const res = await request(app)
      .post('/api/auth/login')
      .send({ email: 'test@test.com', password: 'wrong' });

    expect(res.status).toBe(401);
  });
});

Section 06

GitHub Actions — Two Workflows

You already have deploy.yml for production. You need two more: a CI workflow that tests every PR, and a staging deploy workflow.

File 1 — .github/workflows/ci.yml (NEW)

Runs tests on every PR to staging or main. Blocks merge if tests fail.

name: CI — Test on PR

on:
  pull_request:
    branches: [main, staging]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }

      - name: Install dependencies
        run: npm ci
        working-directory: backend

      - name: Run tests
        run: npm test
        working-directory: backend
        env:
          NODE_ENV: test
          MONGO_URI: ${{ secrets.TEST_MONGO_URI }}
          JWT_SECRET: test-secret-for-ci

File 2 — .github/workflows/staging.yml (NEW)

Auto-deploys to staging when code merges into the staging branch.

name: Deploy to Staging

on:
  push:
    branches: [staging]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to EC2 Staging
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.EC2_HOST }}
          username: ${{ secrets.EC2_USER }}
          key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            cd ~/stratedge-staging
            git fetch origin staging
            git reset --hard origin/staging
            cd backend
            npm install
            pm2 restart stratedge-api-staging
            pm2 restart stratedge-ocr-worker-staging

Section 07

Setting Up Staging on Your EC2

Your EC2 already runs production on port 5000. Staging runs alongside it on port 4000 — same server, separate process, separate database.

On Your EC2 Server — One-Time Setup

# SSH into your EC2
ssh -i your-key.pem ubuntu@your-ec2-ip

# Clone the repo into a separate staging folder
git clone https://github.com/yourusername/stratedge.git ~/stratedge-staging
cd ~/stratedge-staging
git checkout staging

# Create the staging .env file
cp backend/.env backend/.env.staging.backup
nano backend/.env
# Set: PORT=4000, NODE_ENV=staging, MONGO_URI=<staging atlas uri>

# Install dependencies
cd backend && npm install

# Start staging with PM2
pm2 start ecosystem.staging.config.js
pm2 save

ecosystem.staging.config.js — Add to Backend

module.exports = {
  apps: [
    {
      name: "stratedge-api-staging",
      script: "./server.js",
      env: {
        NODE_ENV: "staging",
        PORT: 4000
      }
    },
    {
      name: "stratedge-ocr-worker-staging",
      script: "./workers/ocrWorker.js",
      env: {
        NODE_ENV: "staging"
      }
    }
  ]
};

Nginx — Route staging subdomain to port 4000

# /etc/nginx/sites-available/staging.yourdomain.com
server {
  listen 80;
  server_name staging.yourdomain.com;

  location / {
    proxy_pass http://localhost:4000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
  }
}

The Bottom Line

Right now, every commit you push goes directly to real users with zero verification. After this setup, every commit goes through automated tests → staging environment → manual check → then production. You catch bugs before users do. Your production stays stable. You ship with confidence.

The entire setup takes about 4 hours. After that, the overhead per feature is 5 extra minutes of manual testing on staging. That's a good trade.
